# 02 — Resume Editor: UI Flow

## Routes

| route | screen |
|---|---|
| `/[locale]/profile` | full editor |
| `/[locale]/profile/preview` | "preview as employer" view (read-only) |
| `/[locale]/profile/reupload` | resume re-upload flow with confirmation |

Authenticated worker only. Workers with `onboardedAt = null` are redirected to `/[locale]/onboarding/...` instead.

## Editing pattern

- Tap any field → inline edit mode (input replaces text).
- Blur or press Enter → autosave (`PATCH /v1/profile/...`).
- Status pill ("Saving…" / "Saved" / "Couldn't save — tap to retry") visible at bottom-right.
- Optimistic update: UI reflects the new value immediately; rollback on save failure.

## Card components

### Experience card

```
┌─────────────────────────────────┐
│ Job title         [edit] [✕]    │
│ Employer Name                    │
│ Jan 2024 – Present               │
│ ─                                │
│ • Bullet 1 (tap to edit)         │
│ • Bullet 2                       │
│ [+ Add bullet]                   │
└─────────────────────────────────┘
```

Inline edits per field; Add bullet appends to `bullets[]`; remove icon on each bullet on tap.

### Education / Certification cards

Similar — read-only summary with tap-to-edit fields.

### Skill chips

Multi-select chip strip. Tap a chip to deselect; "+ Add" opens a custom-skill input. Same component as onboarding skills picker (see [01-onboarding/04-ui.md](../01-onboarding/04-ui.md)).

### Availability grid

Same component as onboarding availability. Tap cells to toggle.

## Re-upload flow

`/profile/reupload`:

1. Confirmation screen: "Replacing your resume will overwrite all sections. Your current data will be saved as a backup. Continue?" — Two buttons: Cancel / Replace.
2. On Confirm → file picker (same as onboarding).
3. Spinner with parse status polling.
4. Success → redirect to `/profile` with banner "Resume updated. Review the parsed data."
5. Failure → "Couldn't read this file. Your existing profile is unchanged." → back button.

## Preview as employer

Read-only view rendered from `GET /v1/profile/preview-as-employer`. Shows:

- First name + last initial
- County (not zip)
- Skills
- Experience
- Certifications
- Note: "Your phone and email are only shared after you apply to a job."

A button "Back to edit" returns to `/profile`.

## States

- **Loading:** skeleton matches card layout.
- **Empty section** (e.g., no certifications): show "+ Add certification" placeholder, not an empty card.
- **Save error:** banner persists until manually dismissed or save succeeds.
- **Conflict / stale:** if the server returns a newer `updatedAt` than the client knows, prompt "This profile was updated in another tab. Refresh to see the latest." with a Refresh button. Don't auto-refresh — user might lose work.

## Accessibility

- Same a11y rules as onboarding ([01-onboarding/04-ui.md](../01-onboarding/04-ui.md)).
- Each editable field announces "Editable. Press Enter to save." to screen readers.
- Status pill is `aria-live="polite"`.

## Mobile / PWA

- Single-column layout on mobile.
- Long sections collapsible.
- Service-worker caches the editor shell so it loads offline; edits queue and flush on reconnect.

## Components

- `<ProfileSection>` — collapsible card wrapper
- `<EditableField>` — text/number/date single-field with inline edit
- `<EditableList>` — array of cards with add/remove
- `<SkillChips>`
- `<AvailabilityGrid>`
- `<SaveStatus>`
- `<PreviewBadge>` — visible in preview-as-employer view

All in `packages/ui`.
