# 09 — SEO & AIO: UI Patterns

## Page-level conventions

Every public page MUST include:

1. `generateMetadata` exporting full Metadata object (title, description, canonical, hreflang alternates, OG, Twitter).
2. `<script type="application/ld+json">` with the appropriate schema type, rendered server-side in the page component.
3. Visible NAP (name/address/phone) in the footer (rendered by the shared `Footer` component).
4. `<html lang>` matching the URL locale (handled by `[locale]/layout.tsx`).
5. Crawlable text content — no critical info hidden behind JS-only rendering. Server-render product info, employer name, wage, dates.

## Title and description rules

- **Title length** ≤ 60 chars to avoid Google truncation.
- **Description length** ≤ 160 chars.
- Title format: `<entity name> — <key qualifier> | AgConn` (e.g., "Strawberry Picker — Fresno County | AgConn").
- Description: first sentence is direct-answer (great for AI summaries) — say what the entity is, where, and one key detail.

> **Inferred:** Direct-answer first sentence is an AIO best practice. AI tools tend to quote the first 1-2 sentences. Don't bury the lede with marketing prose.

## Image alt text

- Every `<Image>` has `alt` in BOTH locales via i18n key.
- Employer logos: `alt={t('image.employerLogo', { name })}`.
- Stock imagery: descriptive, not decorative ("Workers harvesting strawberries in a Central Valley field" not "Harvest scene").
- Decorative-only images: `alt=""` and `aria-hidden="true"`.

## Footer NAP component

```tsx
// packages/ui/src/Footer.tsx
export function Footer({ tenant, locale }: { tenant: Tenant; locale: Locale }) {
    return (
        <footer className='bg-base-200'>
            <div className='container'>
                <div itemScope itemType='https://schema.org/Organization'>
                    <span itemProp='name'>AgConn</span>
                    {' • '}
                    <span itemProp='address' itemScope itemType='https://schema.org/PostalAddress'>
                        <span itemProp='streetAddress'>{tenant.settings?.nap?.streetAddress ?? '...'}</span>
                        {', '}
                        <span itemProp='addressLocality'>{tenant.settings?.nap?.locality ?? 'Fresno'}</span>
                        {', '}
                        <span itemProp='addressRegion'>CA</span> <span itemProp='postalCode'>{tenant.settings?.nap?.postalCode ?? ''}</span>
                    </span>
                    {' • '}
                    <span itemProp='telephone'>{tenant.settings?.nap?.phone ?? '+1-559-555-0100'}</span>
                </div>
                <LangToggle />
            </div>
        </footer>
    );
}
```

> **Inferred:** Microdata on the footer NAP supplements the JSON-LD. Some legacy crawlers prefer microdata; AI tools accept both. Use both, lose nothing.

## Hreflang and locale signals

Verified by every page rendering:

```html
<link rel="alternate" hreflang="en" href="https://agconn.com/en/jobs/abc" />
<link rel="alternate" hreflang="es" href="https://agconn.com/es/trabajos/abc" />
<link rel="alternate" hreflang="x-default" href="https://agconn.com/es/trabajos/abc" />
<link rel="canonical" href="https://agconn.com/<active-locale>/..." />
```

Generated automatically by Next when `alternates.languages` is set in metadata.

## Core Web Vitals targets

| metric                          | target           |
| ------------------------------- | ---------------- |
| LCP (Largest Contentful Paint)  | < 2.5s mobile 4G |
| CLS (Cumulative Layout Shift)   | < 0.1            |
| INP (Interaction to Next Paint) | < 200ms          |
| TTFB (Time to First Byte)       | < 800ms          |
| Total Blocking Time             | < 200ms          |

Patterns to maintain these:

- **No CLS:** every image has explicit `width`/`height`. Skeletons match final layout. Web fonts use `next/font` with `display: 'swap'` (or `optional` for non-critical).
- **Fast LCP:** server-render the hero. Don't lazy-load above-the-fold images. Use `priority` on the LCP image.
- **Small JS:** RSC default. `"use client"` only where needed. Tree-shake daisyUI.
- **Edge OG images** so social previews don't bottleneck on the API.

## Lighthouse CI

CI step runs Lighthouse on representative URLs:

```yaml
- run: npm run lighthouse:ci
```

Config at `apps/web/lighthouse.config.js` lists:

- `/es` (landing)
- `/es/jobs`
- `/es/trabajos/<sample-slug>`
- `/es/training`
- `/es/faq`
- Same in EN

Thresholds: SEO ≥ 95, Performance ≥ 80 mobile, Accessibility ≥ 95, Best Practices ≥ 90. Below threshold fails CI.

## Direct-answer paragraph pattern

Top of FAQ pages, jobs listing, training listing:

```tsx
<p className='text-lg max-w-prose'>
    AgConn connects farmworkers in Fresno, Kern, Kings, Madera, and Tulare counties to verified seasonal jobs and CDFA-funded training programs. Workers can sign up with just a phone number; employers
    post jobs after FLC license verification.
</p>
```

This paragraph is the AI-citation target. Keep it factual, concrete, and free of marketing fluff.

## What pages should be `noindex`

- All admin routes
- All API routes (Next.js handles via robots)
- Onboarding and dashboard routes (signed-in only)
- Sign-in / OTP verify pages

Pattern:

```tsx
export const metadata: Metadata = { robots: { index: false, follow: false } };
```
