## Toast

A single queue, mounted once at the root layout via `<ToastProvider>`. The visible region is an `aria-live="polite"` polite container in the bottom-right (desktop) or bottom-center (mobile, above the safe area).

### API

```ts
type ToastVariant = 'success' | 'error' | 'warning' | 'info';

type ToastInput = {
    variant: ToastVariant;
    title: string;                      // already localized at call site
    description?: string;
    durationMs?: number;                // override default
    sticky?: boolean;                   // disables auto-dismiss
    action?: { label: string; onClick: () => void };
    dedupeKey?: string;                 // collapse repeats
};

const toast = useToast();
toast({ variant: 'success', title: t('application.submitted') });
toast({ variant: 'error', title: t('shell.error.offline.title'), sticky: true, dedupeKey: 'offline' });
```

### Defaults

| variant | duration | role        | dismiss         |
| ------- | -------- | ----------- | --------------- |
| success | 5s       | status      | auto + manual   |
| info    | 5s       | status      | auto + manual   |
| warning | 6s       | status      | auto + manual   |
| error   | 8s       | alert       | auto + manual   |

### Behavior

- Maximum 3 visible at once. Newer toasts push older ones into a "+N more" collapsed pill that expands on hover/tap.
- `dedupeKey` collapses repeats — calling toast with `dedupeKey: 'offline'` 5 times shows one toast and resets its timer.
- Hover or focus on a toast pauses its timer (a11y: prefers-reduced-motion users still pause but without the wiggle).
- Tap-to-dismiss target ≥ 44×44 px per [docs/brand/12-accessibility.md](../../brand/12-accessibility.md).
- Mobile bottom-center positioning respects `env(safe-area-inset-bottom)`.

> **Inferred:** Using a custom toast implementation (small, ~150 lines) over a library like sonner or react-hot-toast. Reasons: bilingual is non-negotiable and library defaults often slip English in (e.g., default "Loading..." aria labels); we want dedupe; we want to call `toast()` from the API client without a React hook context (see implementation note below).

### Calling toast from non-React code

The API client lives outside React. To allow it to emit toasts, the toast provider exposes a singleton ref:

```ts
// packages/ui/src/toast/singleton.ts
let push: ((t: ToastInput) => void) | null = null;
export const setToastPusher = (fn: typeof push) => { push = fn; };
export const pushToast = (t: ToastInput) => push?.(t);
```

`<ToastProvider>` calls `setToastPusher` on mount. The API client provider calls `pushToast` from its `onUnhandledError` hook. This keeps the API client React-agnostic.

## Modal & confirm

```ts
type ModalProps = {
    open: boolean;
    onClose: () => void;
    title: string;
    description?: string;
    size?: 'sm' | 'md' | 'lg';
    children: React.ReactNode;
};

type ConfirmInput = {
    title: string;
    description?: string;
    confirmLabel?: string;            // default t('shell.modal.confirm')
    cancelLabel?: string;             // default t('shell.modal.cancel')
    variant?: 'default' | 'destructive';
};

// imperative
const confirmed = await confirm({ title: t('jobs.delete_title'), variant: 'destructive' });
if (!confirmed) return;
```

### Behavior

- Focus trap on open; first focusable element receives focus.
- ESC closes (calls `onClose`) unless `variant === 'destructive'` requires explicit click.
- Click on backdrop closes (same rule).
- Returns focus to the trigger element on close.
- daisyUI `<dialog>` element under the hood with custom Tierra styling (no default rounded corners, brand color outline).
- `confirm()` returns `Promise<boolean>`; `alert()` returns `Promise<void>`.

## Form

The form primitive wires `react-hook-form` with `@hookform/resolvers/zod` and integrates the API client error envelope into field-level errors.

### API

```tsx
const schema = z.object({
    email: z.string().email(),
    fullName: z.string().min(1),
});

type FormValues = z.infer<typeof schema>;

<Form
    schema={schema}
    defaultValues={{ email: '', fullName: '' }}
    onSubmit={async (values, ctx) => {
        const res = await api.post<Worker>('/v1/onboarding', values, { handleErrorInline: true });
        if (!res.ok) {
            if (res.error.fields) ctx.setErrors(res.error.fields);
            return;
        }
        toast({ variant: 'success', title: t('onboarding.saved') });
        router.push('/dashboard');
    }}
>
    <Field name="fullName" label={t('field.full_name.label')} />
    <Field name="email" label={t('field.email.label')} type="email" />
    <FormSubmit>{t('action.continue')}</FormSubmit>
</Form>
```

### Behavior

- `<Field>` wires `register(name)` automatically and renders label + input + `<FieldError>`.
- `<FieldError>` reads from `formState.errors[name]` AND from server-side errors set via `ctx.setErrors(fields)`. Server errors clear when the user edits the field.
- `<FormSubmit>` is disabled while `formState.isSubmitting` and shows a spinner; double-tap protected.
- Validation triggers on blur for the first attempt, then on change after the first error (RHF default). Submission revalidates everything.
- All zod issues map to localized messages via the `shell.validation.*` namespace.

## ErrorBoundary

Two flavors:

```tsx
// component-level
<ErrorBoundary fallback={<InlineError />}>
    <FlakySection />
</ErrorBoundary>

// page-level (used in app/[locale]/error.tsx)
<PageError reset={reset} digest={error.digest} />
```

`<PageError>` shows a Tierra-themed friendly fallback with:
- a localized title + description
- a "Try again" button (`reset()`)
- a small "Error ID: {digest}" line so support can correlate with Sentry
- a link back to home

Reports to Sentry on mount with the React component stack.

## Page errors (404/500)

```tsx
// app/[locale]/not-found.tsx → <PageError variant="404" />
// app/[locale]/error.tsx     → <PageError variant="500" reset={reset} digest={...} />
```

Both render inside the same shell as authenticated pages so navigation is preserved.

## PWA wrapper

The PWA wrapper is what makes the worker app installable and resilient on patchy LTE. Lives in `apps/web` (it's app-specific, not a shared package).

### Manifest

> **Inferred:** Using `dir: "ltr"` and a single manifest for both EN and ES. Both languages are LTR. Localized `name` / `description` per-locale would require either two manifests (linked via `<link rel="manifest" hreflang>`) or a single manifest with the dominant locale. Theme color `#A85A2A` is the Tierra primary. Background `#FFFCF7` is the Tierra paper. Both pulled from [docs/brand/02-color.md](../../brand/02-color.md) — keep in sync.

### Service worker — serwist

The PWA is built on **[serwist](https://serwist.pages.dev/)** (`@serwist/next`). serwist is a workbox-derived library purpose-built for Next 15 App Router and is the recommended path forward in this project. Reasons:

- Native Next 15 App Router support (next-pwa is community-maintained and lags Next releases).
- TypeScript-first config; precache manifest is generated at build time and typed.
- First-class support for runtime caching strategies via composable `Strategy` objects.
- Works cleanly with Turbopack and the new build pipeline.

Setup:

```ts
// apps/web/src/app/sw.ts
import { defaultCache } from '@serwist/next/worker';
import { Serwist } from 'serwist';

declare const self: ServiceWorkerGlobalScope & { __SW_MANIFEST: (string | { url: string; revision: string | null })[] };

const serwist = new Serwist({
    precacheEntries: self.__SW_MANIFEST,
    skipWaiting: false,                     // explicit refresh per "update toast" UX
    clientsClaim: false,
    navigationPreload: true,
    runtimeCaching: [
        ...defaultCache,
        // override / extend per the strategy table below
    ],
    fallbacks: {
        entries: [
            { url: '/en/offline', matcher: ({ request }) => request.destination === 'document' },
            { url: '/es/offline', matcher: ({ request }) => request.destination === 'document' },
        ],
    },
});
serwist.addEventListeners();
```

```ts
// apps/web/next.config.ts (excerpt)
import withSerwistInit from '@serwist/next';

const withSerwist = withSerwistInit({
    swSrc: 'src/app/sw.ts',
    swDest: 'public/sw.js',
    cacheOnNavigation: true,
    reloadOnOnline: false,
    disable: process.env.NODE_ENV === 'development',
    exclude: [/^\/admin\//, /^\/admin\/v1\//, /^\/llms\.txt$/, /^\/sitemap\.xml$/, /^\/robots\.txt$/],
});

export default withSerwist({ /* rest of next config */ });
```

Strategy:

| pattern                          | strategy            | notes                                                   |
| -------------------------------- | ------------------- | ------------------------------------------------------- |
| `/_next/static/*`                | cache-first         | Hashed assets, immutable                                |
| `/icons/*`, `/fonts/*`, `/img/*` | cache-first         | Long max-age                                            |
| `/`, `/[locale]/*` (HTML)        | network-first       | Falls back to `/[locale]/offline` cached page           |
| `/api/*`                         | network-only        | Mutations and live data; do not cache                   |
| `/admin/*`, `/admin/v1/*`        | **not registered**  | Out of SW scope entirely                                |
| `/llms.txt`, `/sitemap.xml`, `/robots.txt` | network-only | Crawlers see fresh content                       |

The offline fallback page lives at `/[locale]/offline/page.tsx` and is precached on install.

### Install prompt UX

```tsx
// packages/ui/src/pwa/install-prompt.tsx
'use client';
export function InstallPrompt() {
    const [evt, setEvt] = useState<BeforeInstallPromptEvent | null>(null);
    const [dismissed, setDismissed] = useLocalStorage('pwa.install.dismissed', false);
    const t = useTranslations('shell.install');

    useEffect(() => {
        const handler = (e: Event) => {
            e.preventDefault();
            setEvt(e as BeforeInstallPromptEvent);
        };
        window.addEventListener('beforeinstallprompt', handler);
        return () => window.removeEventListener('beforeinstallprompt', handler);
    }, []);

    if (!evt || dismissed) return null;
    return (
        <Card>
            <h3>{t('title')}</h3>
            <p>{t('body')}</p>
            <Button onClick={async () => { await evt.prompt(); setEvt(null); }}>{t('cta')}</Button>
            <Button variant="ghost" onClick={() => setDismissed(true)}>{t('later')}</Button>
        </Card>
    );
}
```

- Shows only when the browser fires `beforeinstallprompt`.
- Hidden if previously dismissed (localStorage flag, no expiry — the user opted out).
- Hidden if `display-mode: standalone` (already installed).
- iOS-specific fallback: Safari does not fire `beforeinstallprompt`. A `<IosInstallHint>` appears once per device showing "Tap Share → Add to Home Screen" with an inline icon. Tracks dismissal separately.

Placement: in the worker dashboard sidebar / mobile drawer, never on the marketing landing.

### Offline banner

Sticky `aria-live="polite"` banner shown when `navigator.onLine === false`:

```tsx
// packages/ui/src/pwa/offline-banner.tsx
'use client';
export function OfflineBanner() {
    const [online, setOnline] = useState(true);
    useEffect(() => {
        const update = () => setOnline(navigator.onLine);
        update();
        window.addEventListener('online', update);
        window.addEventListener('offline', update);
        return () => {
            window.removeEventListener('online', update);
            window.removeEventListener('offline', update);
        };
    }, []);
    if (online) return null;
    const t = useTranslations('shell.offline');
    return (
        <div role="status" aria-live="polite" className="sticky top-0 bg-warning text-warning-content py-2 px-4 text-sm">
            {t('banner')}
        </div>
    );
}
```

The corresponding offline page (`/[locale]/offline`) shows a longer message + cached "saved jobs / saved applications" if available (Tier 2; for MVP, a static message).

### Update prompt

When a new service worker is waiting, show a non-blocking toast:

```ts
// in the SW registration
wb.addEventListener('waiting', () => {
    pushToast({
        variant: 'info',
        title: t('shell.update.title'),
        description: t('shell.update.description'),
        sticky: true,
        dedupeKey: 'sw-update',
        action: {
            label: t('shell.update.cta'),
            onClick: () => {
                wb.messageSkipWaiting();
                window.location.reload();
            },
        },
    });
});
```

The user is never force-reloaded. If they ignore the toast, the new SW activates on next idle navigation.

## Mounting order

The root layout composes providers top-down. Order matters — providers below depend on providers above.

```tsx
// apps/web/src/app/[locale]/layout.tsx
<NextIntlClientProvider locale={locale} messages={messages}>
    <ToastProvider>
        <ModalProvider>
            <ApiProvider baseUrl={env.NEXT_PUBLIC_API_BASE}>
                <ClerkProvider>
                    <ServiceWorkerProvider />        {/* registers SW, listens for updates */}
                    <OfflineBanner />
                    {children}
                    <InstallPrompt />
                </ClerkProvider>
            </ApiProvider>
        </ModalProvider>
    </ToastProvider>
</NextIntlClientProvider>
```

Marketing routes (`/en`, `/es`) skip `<ApiProvider>`, `<ClerkProvider>`, `<ServiceWorkerProvider>`, and `<InstallPrompt>` — the marketing layout is leaner.

## Accessibility

- All primitives meet WCAG 2.1 AA per [docs/brand/12-accessibility.md](../../brand/12-accessibility.md).
- `aria-live` regions: toast (`polite`), offline banner (`polite`), error boundary inline (`assertive`).
- Focus management: modal traps focus, returns on close; toast does not steal focus.
- Reduced motion: all primitives honor `prefers-reduced-motion: reduce` (toast slides become fades, modal scale becomes opacity).
- Touch targets ≥ 44×44 px on all interactive surfaces.
- High-contrast: uses Tierra theme tokens which already meet contrast in both light and dark variants.

## Mobile / PWA notes

- Tap targets ≥ 44 px.
- Bottom-fixed elements (offline banner, toast region) respect `env(safe-area-inset-bottom)`.
- The install prompt + iOS hint are tested on iOS Safari, Chrome Android, and standalone (post-install) modes.
- The offline page is precached and bilingual; both `/en/offline` and `/es/offline` are in the precache manifest.
- The worker app is the primary PWA target; employer/admin dashboards are usable but not optimized for offline.
