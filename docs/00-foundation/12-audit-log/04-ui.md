# 12 — Audit Log: UI (Admin Viewer)

The audit viewer is a **read-only admin surface**. It lives at `/admin/audit`. End users (workers, employers, training orgs) never see this UI in MVP.

The viewer is **rendered in English at MVP but built bilingual-by-design** — every visible string reads from `messages/{en,es}.json` under `admin.audit.*`. The Spanish file carries the same key shape with empty values during MVP and is filled in by the Spanish reviewer post-MVP without any code changes. See [05-i18n.md](05-i18n.md) for the full namespace and the parity-check allowlist.

## Routes

```
/admin/audit                              # main timeline (default: this tenant, last 7 days)
/admin/audit/event/[id]                   # single event detail
/admin/audit/correlation/[correlationId]  # all events sharing a correlation id
/admin/audit/actor/[actorId]              # actor timeline (worker / employer / admin)
/admin/audit/resource/[type]/[id]         # resource history
/admin/audit/export                       # filter form → CSV download
/admin/audit/redact                       # CCPA redaction tool (gated, second-factor)
```

## Main timeline (`/admin/audit`)

### Layout

A standard admin shell (top nav, side rail, main content) with three regions:

1. **Filter bar** at the top.
2. **Timeline** as the main content — virtualized list, infinite scroll.
3. **Detail drawer** on the right when a row is selected (slides in, doesn't navigate).

```
┌──────────────────────────────────────────────────────────────────┐
│  Filters: [Tenant ▼] [Actor ▼] [Action ▼] [Outcome ▼] [Date ▼] │
├──────────────────────────────────────────────────────────────────┤
│  09:23  worker.application.submitted  worker_42 → job_8         │
│  09:21  auth.login                     worker_42                 │
│  09:18  job.posting.created            employer_3 → job_8       │
│  ...                                                             │
└──────────────────────────────────────────────────────────────────┘
```

### Filter bar

Filters are URL-driven (every filter state is shareable / bookmarkable):

- **Tenant** — defaults to caller's primary tenant. Dropdown lists every tenant the admin has access to. Required.
- **Actor** — typeahead search by `actorId` or by user name (joined client-side from `/admin/v1/users` endpoint). Multi-select.
- **Action** — multi-select dropdown organized by domain. Each option shows the action code in monospace (`auth.login`, `job.posting.created`) plus an English description from a static map. Includes a "prefix mode" toggle (`auth.*`).
- **Resource** — typeahead `resourceType:resourceId` (e.g., `application:01HM...`).
- **Outcome** — `success` / `failure` / both (default both).
- **Date range** — preset chips (1h, 24h, 7d, 30d, custom). Default: 7d.
- **Correlation id** — paste a UUID; collapses other filters.
- **Reset** — clears all filters back to defaults.

Filter changes update the URL (`pushState`) and refetch.

### Timeline rows

Each row shows:

```
┌─ Time ──┬─ Action ──────────────────────┬─ Actor ────────────┬─ Resource ───────────┬─ Outcome ─┐
│ 09:23   │ worker.application.submitted  │ worker_42 (worker) │ application:01HM... │ ✓ success │
└─────────┴───────────────────────────────┴────────────────────┴──────────────────────┴───────────┘
```

- Time is in admin's local timezone (toggleable to UTC).
- Action shown in monospace + a friendly description on hover.
- Actor links to `/admin/audit/actor/[id]`.
- Resource links to `/admin/audit/resource/[type]/[id]`.
- Failure outcome rendered with Tierra `error` token (red text, bold).
- Click anywhere on a row opens the detail drawer.
- Right-click → context menu with "Copy event id", "Copy correlation id", "Open correlation timeline", "Copy as JSON".

### Virtualization & pagination

`react-window` for virtualization. The list pages via the cursor returned from `/admin/v1/audit/events`:

- Initial fetch: 50 rows.
- Scroll near bottom: fetch next 50 with `cursor=...`.
- Scroll near top after a previous page: re-fetch with the previous cursor (no anchor scroll loss).
- Refresh button + auto-refresh-every-30s toggle (off by default — admins prefer stable views during incident triage).

### Empty / loading / error

- Empty: `<EmptyState>` with "No events match your filters."
- Loading first page: `<PageSkeleton variant="timeline" />`.
- Loading next page: small inline spinner at the bottom.
- Error: full-pane `<PageError variant="500" />` with retry.

### Performance UX

For the default (`tenant_id` + 7d) the page should TTI in under 1s on a desktop with 100M-row table.

For unfiltered super-admin queries, show a non-blocking warning banner: "This view spans all tenants and may be slow. Add a tenant filter to speed it up."

Metadata renders as syntax-highlighted JSON. Long strings are clipped with click-to-expand.

## Correlation timeline (`/admin/audit/correlation/[id]`)

Shows every event sharing the correlation id, in chronological order. This is the most-used incident-response view: "the user clicked submit and got an error — what actually happened?"

Layout: vertical timeline with each event as a card. The first event (typically the inbound HTTP request, when we add request audit) and the last event (typically the response or error) are highlighted.

If any event in the chain has `outcome: 'failure'`, render a banner at the top: `"This request chain ended in failure: error.unhandled"`.

## Actor timeline (`/admin/audit/actor/[id]`)

A vertical timeline of recent activity for one actor. Used for support: "the worker says they applied but it's not in the system — what happened?"

Pinned at the top: the actor's basic profile (name, role, tenant, last login, account status — fetched from `/admin/v1/users/:id`).

The timeline groups events by date heading (Today, Yesterday, Earlier this week, May 2026, ...).

Cross-references: each event links to its correlation timeline and to the resource it touched.

## Resource history (`/admin/audit/resource/[type]/[id]`)

Every event that touched a resource, oldest-first (so you read top-to-bottom as a story). Used for: "show me this application's lifecycle from submit to hire."

For known resource types, render a header card with current state pulled from the live API:

- `application` → status, applicant, employer, job
- `job_posting` → title, employer, status
- `worker_profile` → name, county, current job
- `tenant` → slug, name, plan

Diff view: when consecutive events both touch the same resource and report a `fields` array in metadata (e.g., `worker.profile.updated`), show a delta panel between them.

## Export (`/admin/audit/export`)

Form that mirrors the timeline filters and produces a CSV:

```
[Tenant ▼] [Actor ▼] [Action ▼] [Outcome ▼] [Date ▼]

[Estimated rows: 1,247]   [Download CSV]
```

The estimate hits a `/admin/v1/audit/events/count?...` endpoint that runs an EXPLAIN-ANALYZE-style estimator (uses Postgres `pg_stats`) — fast even for huge ranges.

The download button:

1. Fires `GET /admin/v1/audit/export?...` (streamed CSV).
2. Browser saves as `agconn-audit-{tenantSlug}-{from}-{to}.csv`.
3. The endpoint emits `admin.data.exported` so the export itself is auditable.

Exports above 1M rows are rejected with a hint to narrow the filters.

## Redaction tool (`/admin/audit/redact`)

Heavily gated. Visible only to a dedicated `super_admin` Clerk role (a stricter superset of `admin`). Workflow:

1. Operator enters a `tenantId`, `actorId`, CCPA request id, and selects a reason.
2. UI calls `POST /admin/v1/audit/redact?dryRun=true` to count matching rows.
3. UI shows a confirmation: "This will redact actor PII from {count} events. This is irreversible."
4. Operator clicks confirm; UI requests a second-factor confirmation token via a separate flow (TOTP / Clerk MFA challenge).
5. UI calls `POST /admin/v1/audit/redact` with the `x-confirm` header.
6. Result page shows `{ redactedCount }` and links to the `admin.audit.redacted` event for traceability.

Visual treatment: the entire page is on a `bg-error-content` background tint with a "Destructive operation" banner so no operator confuses it with the read-only timeline.

## Accessibility

- All interactive elements meet WCAG 2.1 AA (Tierra default).
- Timeline rows are keyboard-navigable (arrow keys move selection; Enter opens drawer; Esc closes).
- Filter dropdowns: `combobox` ARIA pattern with `aria-controls`, `aria-expanded`.
- Detail drawer is a `dialog` with focus trap and proper `aria-labelledby`.
- Failure rows have an icon in addition to color so they're distinguishable for color-blind users.

## Mobile / responsive

The audit viewer is **desktop-first**. On screens < 768 px:

- Filter bar collapses into a single "Filters" button that opens a full-screen sheet.
- Timeline rows stack the columns vertically.
- Detail drawer becomes a full-screen takeover.
- Export / redact pages remain accessible but are practically only used on desktop.

Admins are not the primary mobile audience; this is a triage / support tool used at a desk.

## Performance contracts

- Initial page load: < 1.5 s on broadband, including first batch of 50 rows.
- Filter change: < 300 ms perceived (skeleton in 50 ms; data in < 300 ms with `tenant_id + 7d`).
- Detail drawer open: instant (data already in row).
- Correlation timeline: < 500 ms (typically 1–6 rows).
- Export: streams immediately; does not block the UI.
