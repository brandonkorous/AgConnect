# 12 — Audit Log: i18n

The audit log itself has **no end-user-facing copy**. Workers, employers, and training orgs never see audit content in MVP. The admin viewer is rendered in English for MVP — but the data layer and the rendering layer are **bilingual by design** so dropping in Spanish translations later is a copy-paste, not a refactor.

## Design rule: every visible string lives in `messages/{en,es}.json`

No literal English strings in `apps/web/src/app/[locale]/admin/audit/` or `packages/audit/`. Every label, description, column header, filter option, button, and empty-state message is read via `useTranslations('admin.audit.*')`. The Spanish file (`es.json`) carries the same key shape with empty values during MVP; the parity-check allowlist permits this temporary state.

When the Spanish reviewer arrives, they fill in `es.json` keys and the admin viewer becomes bilingual without touching any TS, JSX, or routing code.

## CI parity check — admin allowlist

The parity check (`packages/i18n/scripts/check-parity.ts`) is extended with an allowlist of namespaces where empty ES values are permitted during MVP:

```ts
// packages/i18n/scripts/check-parity.ts (excerpt)
const EMPTY_ES_ALLOWLIST = [
    'admin.audit.*',          // tracked at https://… — Spanish admin viewer post-MVP
    // add other admin namespaces here as they appear
];

function isAllowedEmpty(key: string) {
    return EMPTY_ES_ALLOWLIST.some((pattern) => matchesGlob(key, pattern));
}
```

The check still asserts:

1. Every key present in `en.json` is present in `es.json` (shape parity).
2. Every value in `en.json` is non-empty.
3. Every value in `es.json` is non-empty **unless** the key is allowlisted.
4. Every interpolation token in EN appears in ES (when ES has a value).

When the Spanish admin reviewer signs off, the allowlist entries get removed and the check enforces real parity going forward. The diff is small and reviewable.

## Renderer rule

The audit viewer renders an action with:

```tsx
const t = useTranslations('admin.audit');
// fall back to en when es is empty
const label = t(`action.${event.action}.label`, { default: tEn(`action.${event.action}.label`) });
```

`tEn` is a server-side translator pinned to English for the fallback. Works because we always have English; we don't always have Spanish during MVP.

> The English-fallback pattern keeps the viewer usable in `/es/admin/audit` without machine-translating audit content. When Spanish lands, the fallback never fires.

## End-user-facing audit-derived emails

When end users receive emails about audit-related operations (CCPA exports, account deletion, redaction confirmations), those emails ARE bilingual at MVP — they go to workers and employers who use Spanish daily. Strings live in their feature folders:

- CCPA data export confirmation → `data_export.*` (future feature)
- Account deletion notice → `account.deleted.*` (covered in worker/employer onboarding folders)
- Redaction confirmation to affected user → `data_export.redaction.*` (future feature)

The audit row itself stays English-coded; the email template renders in the recipient's locale.

## Storage rule (unchanged)

The `metadata` column stores English-only structured data — codes, ids, action strings, never localized text. Rendering happens in the consumer's locale at view time. This rule was load-bearing before bilingual-by-design and remains so: it lets us add Spanish (or any locale) post-hoc without migrations.

Bad:

```jsonc
{ "errorMessage": "Hubo un problema al guardar tu perfil" }
```

Good:

```jsonc
{ "errorCode": "internal_error", "route": "/v1/worker/profile", "method": "PATCH" }
```

## Spanish reviewer checklist (when ready)

1. Open `packages/i18n/messages/es.json`.
2. Translate every `admin.audit.*` value, following the [Tierra voice guide](../../brand/04-voice.md): tuteo (`tú`), short sentences, no anglicisms.
3. Pay special attention to `action.<code>.label` strings — these appear in dense table rows; keep them under 30 chars where possible.
4. Run `pnpm i18n:check-parity` locally to confirm zero allowlist hits.
5. Remove the `admin.audit.*` entry from `EMPTY_ES_ALLOWLIST` in `packages/i18n/scripts/check-parity.ts`.
6. Open the admin viewer at `/es/admin/audit` and review every screen.
7. Submit one PR with all of the above; the diff is contained to two files.
