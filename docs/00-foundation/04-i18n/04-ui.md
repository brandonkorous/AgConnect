# 04 — i18n: UI Patterns

## Locale-prefixed app structure

```
apps/web/app/
  [locale]/
    layout.tsx               # NextIntlClientProvider + Clerk
    page.tsx                 # landing
    jobs/                    # EN: /en/jobs/...
    trabajos/                # ES alias  /es/trabajos/...
    onboarding/...
  api/                       # locale-agnostic
    health/route.ts
  middleware.ts
  i18n.ts
```

`[locale]` is statically typed; only `'en' | 'es'` are valid.

For top-level alias routes (`jobs` ↔ `trabajos`), use Next's rewrite or a single canonical path with hand-managed redirects. **Default for MVP: rewrite via `next.config.ts` `redirects()`** so both URLs resolve to the same handler.

## Lang toggle placement

- Welcome / sign-in screens: large center-of-page toggle.
- Authenticated pages: top-right, in the user menu; small `EN | ES` join-buttons.
- Footer: also surfaces the toggle for users who didn't notice the header.

## Layout reads

```tsx
// apps/web/app/[locale]/layout.tsx
import { NextIntlClientProvider } from 'next-intl';
import { getMessages } from 'next-intl/server';

export default async function LocaleLayout({
  children,
  params: { locale },
}: {
  children: React.ReactNode;
  params: { locale: 'en' | 'es' };
}) {
  const messages = await getMessages();
  return (
    <html lang={locale} dir="ltr">
      <body>
        <NextIntlClientProvider locale={locale} messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

## Server component usage

```tsx
import { getTranslations } from 'next-intl/server';

export default async function Page() {
  const t = await getTranslations('worker.onboarding.welcome');
  return <h1>{t('tagline')}</h1>;
}
```

## Client component usage

```tsx
'use client';
import { useTranslations } from 'next-intl';

export default function Form() {
  const t = useTranslations('worker.onboarding.phone');
  return <label>{t('label')}</label>;
}
```

## Bilingual content rendering (DB-sourced)

```tsx
// for content from the DB with _en/_es columns
function Title({ posting, locale }: { posting: JobPosting; locale: Locale }) {
  return <h2>{locale === 'es' ? posting.titleEs : posting.titleEn}</h2>;
}
```

Helper to reduce boilerplate:

```ts
// packages/ui/src/intl.ts
export const localized = <T extends Record<string, any>>(obj: T, field: string, locale: Locale) =>
  obj[`${field}${locale === 'es' ? 'Es' : 'En'}`];
```

## Forms and validation messages

- Form labels and helpers come from `t()`.
- Server-side validation errors come back as i18n keys (e.g., `common.validation.required`); client maps to `t(error.message)` to render.
- Inline errors aria-describe the input.

## Dates, times, currency

Use `Intl.NumberFormat` and `Intl.DateTimeFormat` with the active locale:

```ts
const fmt = new Intl.NumberFormat(locale === 'es' ? 'es-MX' : 'en-US', {
  style: 'currency',
  currency: 'USD',
});
fmt.format(18.5); // "$18.50" / "$18.50"  (currency same; locale affects digit separators)
```

> **Inferred:** `es-MX` for Mexican Spanish formatting matches the user base. For dates, prefer `Intl.DateTimeFormat` with `dateStyle: 'medium'` so output reads naturally in both locales.

## Accessibility

- `<html lang>` set per locale (above).
- Screen reader announces locale-correct text.
- LangToggle buttons have `aria-pressed` reflecting current locale.

## Testing

- Storybook stories include both EN and ES variants for every component with text.
- Playwright tests run twice — once with `LOCALE=en`, once with `LOCALE=es` — for screens with significant text rendering.
