# 03 — Applicant Review: UI

## Routes

| route                                    | screen                                                            |
| ---------------------------------------- | ----------------------------------------------------------------- |
| `/[locale]/employer/dashboard`           | dashboard with read-only kanban summary section                   |
| `/[locale]/employer/inbox`               | cross-posting applicant inbox (list mode)                         |
| `/[locale]/employer/jobs/:id/applicants` | per-posting kanban with job context chrome (breadcrumb, progress) |
| `/[locale]/employer/applications/:id`    | applicant detail drawer / page                                    |

A cross-job pipeline page (`/employer/pipeline`) is deferred — see [09-future-considerations.md](09-future-considerations.md).

## Dashboard summary (read-only)

The dashboard renders a compact pipeline section above the rest of the page. Four lanes (Applied / Reviewed / Hired / Archived), top 3–5 cards per lane, no drag, no bulk select. The section header links into either the inbox (cross-job triage) or — when a posting is in view — the per-posting kanban.

This surface is informational only. To act on a card, the employer is moved to the per-posting kanban or applicant detail. Counts and card shape match the per-posting board so the summary is a faithful preview.

Implementation: existing `<HiringPipelineBoard>` ([apps/web/src/components/employer/dashboard/HiringPipelineBoard.tsx](../../../apps/web/src/components/employer/dashboard/HiringPipelineBoard.tsx)).

## Per-posting kanban

At `/employer/jobs/:id/applicants`. The working surface for stage management on one posting:

```
┌─────────────┬─────────────┬─────────────┐
│  Applied 12 │ Reviewed 4  │  Hired 1    │
│  ┌────────┐ │  ┌────────┐ │  ┌────────┐ │
│  │ Maria  │ │  │ Juan   │ │  │ Pedro  │ │
│  │ Fresno │ │  │ Fresno │ │  │ Fresno │ │
│  │ 3 mtch │ │  │ 2 mtch │ │  │ Hired  │ │
│  └────────┘ │  └────────┘ │  └────────┘ │
│  ...        │             │             │
└─────────────┴─────────────┴─────────────┘
[Show rejected (2)]
```

Chrome above the board (already implemented): back-to-/jobs breadcrumb, eyebrow with title + `#humanId`, wage range, `hireCount/positionsTotal` progress.

Behavior:

- **Drag-and-drop** to move (desktop). Optimistic UI; revert + toast on failure (per [08-edge-cases.md](08-edge-cases.md) "Kanban drag latency").
- **Mobile**: tap card → action sheet ("Mark Reviewed", "Hire", "Reject").
- Dropping into **Hired** opens the [Hire modal](#hire-modal) (wage + start-date capture). Tapping Hire from the action sheet opens the same modal.
- Dropping into **Reviewed** transitions silently (worker SMS + email fires per [10-worker/04-application-tracker/06-messaging.md](../../10-worker/04-application-tracker/06-messaging.md)).
- Rejection is via action sheet or detail page, not via lane drop (Rejected is collapsed under `[Show rejected]`).
- **Keyboard**: Tab to card, Space to grab, arrows to move, Space to drop. @dnd-kit handles this.
- Match label shows skill-match count vs. the job's required skills.
- "Bulk select" checkbox per card → bulk action bar appears at bottom.

## Bulk actions

Bottom action bar appears when 2+ cards selected:

```
[5 selected] [Mark Reviewed] [Reject] [Cancel]
```

Reject → bulk reject modal with ONE reason for all.

## States

- **Empty inbox:** "No applicants yet. Share your posting on social — agconn.com/jobs/<slug>"
- **Posting not yet published:** "Publish this posting to start receiving applicants. [Publish]"
- **Hired all positions:** banner on kanban: "All positions filled. Posting auto-closed." with un-close action (creates a new posting — Phase 2; for MVP, manual reopen by admin).

## Performance

- Inbox is RSC; cards rendered server-side.
- Kanban is client-rendered (drag-and-drop interaction).
- Application detail RSC + client island for status actions.

## Components

- `<EmployerInbox>`, `<ApplicantKanban>`, `<ApplicantCard>`, `<ApplicantDetail>`, `<HireModal>`, `<RejectModal>`, `<BulkActionBar>` — in `apps/web/app/[locale]/employer/applications/_components/`.

## Accessibility

- Drag-and-drop has keyboard alternative (Tab to card, Space to grab, arrows to move, Space to drop). React Beautiful DnD or @dnd-kit handle this.
- Status changes announce via `aria-live`.
